# PRD 001 — Effectum: Agent Teams Integration

## 1. Problem Statement

Effectum's current autonomous workflow (Plan → TDD → Ralph Loop → Verify) is built around a single Claude Code session. The session executes tasks sequentially, switching context between frontend, backend, tests, and documentation. For small-to-medium features this is acceptable. For complex, cross-cutting features — schema migrations, new API surfaces, multi-component UI overhauls, security audits — it is the wrong shape of work.

**The sequential bottleneck has three real costs:**

1. **Context thrash.** A single session context mixes frontend conventions, API logic, SQL migrations, and test harness details. Each domain shift requires re-loading context from scratch or carrying a growing, diluted system prompt. The more layers involved, the worse the signal-to-noise ratio.

2. **No parallelism where parallelism is safe.** Once a schema migration is applied, the API layer and UI layer could proceed in parallel — they touch different files. A single session serializes them anyway. On a fullstack feature this means 2–3x longer wall-clock time with no correctness benefit.

3. **No specialization.** A generalist session reviewing its own security posture is inherently limited. A dedicated security reviewer teammate — with the right prompt, tools, and focus — catches vulnerabilities the generalist misses. The same applies to test coverage, database performance, and accessibility.

Claude Code Agent Teams (shipped experimentally with Opus 4.6, March 2026) solves exactly this. Multiple Claude Code instances collaborate on the same project — sharing a task list, messaging each other directly, and operating in parallel within safe file-ownership boundaries.

**Effectum does not yet integrate Agent Teams.** Users who attempt to use them get no guidance: no team profiles, no `/orchestrate` command, no cost estimation, no hooks, no documentation. The feature exists but is invisible.

**This PRD specifies the full first-class integration** so that Effectum users can invoke teams naturally — with the same plan-first discipline, quality gates, and cost awareness that defines the rest of the framework.

## 3. Out of Scope

The following are explicitly **not** part of this feature:

- **Real-time team dashboard**: A terminal UI or web-based dashboard showing live teammate status. State tracking is file-based (`team-state.local.md`); visual dashboards are a separate project.
- **Automatic merge conflict resolution**: When two teammates produce conflicting edits (despite file ownership rules), the lead must resolve manually. Automated merge is not in scope.
- **Custom subagent YAML tooling**: A CLI command to scaffold `.claude/agents/*.md` files. The format is documented; file creation is manual or via `effectum install`.
- **Cross-project team orchestration**: Teammates running in different repositories. All teammates share the same repo root.
- **Ruflo / Gas Town / external orchestration integration**: Compatibility with third-party orchestration frameworks. Effectum stays native.
- **BMAD or GSD methodology support**: Effectum's own workflow (Plan → TDD → Ralph Loop) is the only supported methodology.
- **Split-pane / tmux mode**: Advanced terminal display modes. The default `in-process` mode is the only supported configuration.
- **Token usage metering API**: Programmatic access to per-teammate token counts. Usage is estimated, not measured.
- **Any changes to the Ralph Loop, TDD, or Verify commands**: This PRD does not alter existing workflow commands.
- **Multi-LLM support**: Teammates run Claude only (Sonnet default, Opus for lead). No Ollama, GPT-4, or Gemini teammates.

## 4. Acceptance Criteria

### Team Profile YAML Format

**AC-1 — Profile YAML schema is valid and loadable**
Given a profile file at `system/teams/{name}.yaml`, when `/orchestrate {name}` is invoked, then the command must parse the YAML without error and extract `name`, `description`, `teammate_count`, `teammates[]`, `phases[]`, and `cost_estimate` fields. A profile with any missing required field must fail with a descriptive error naming the missing field.

**AC-2 — Agent frontmatter fields are respected**
Given a subagent definition at `.claude/agents/{role}.md` with frontmatter fields `name`, `description`, `tools`, `model`, `permissionMode`, `maxTurns`, `isolation`, `memory`, and `hooks`, when a teammate is spawned using that agent definition, then Claude Code must apply all specified frontmatter fields to the spawned instance. A unit test must verify that the frontmatter template produced by `effectum install` is valid YAML and contains all required fields (`name` and `description` are mandatory; all others are optional with documented defaults).

**AC-3 — Custom team profiles are supported**
Given a file at `.effectum/teams/{name}.yaml` (project-local override), when `/orchestrate {name}` is invoked, then the local profile must take precedence over the system profile of the same name. The command must search `.effectum/teams/` first, then `system/teams/`, and report which source was used in the cost estimate output.

---

### Recommendation Engine

**AC-4 — RE recommends Agent Teams for complex cross-domain work**
Given a PRD with ≥10 Acceptance Criteria that span ≥2 distinct file domains (e.g., ACs touching both `src/components/` and `src/app/api/`), when the Recommendation Engine evaluates the project, then it must include `agent-teams` as a suggested workflow in its output, with a justification string explaining the domain count and AC count. The recommendation must specify the matching profile by name (e.g., `web-feature` or `fullstack`).

**AC-5 — RE recommends subagents for focused parallel tasks**
Given a PRD with 4–9 ACs that are independent but touch a single domain (e.g., all ACs are in `src/components/`), when the Recommendation Engine evaluates the project, then it must recommend `subagents` (not Agent Teams) and explain that coordination overhead would outweigh the benefit of peer-to-peer messaging.

**AC-6 — RE recommends single session for sequential work**
Given a PRD with ≤5 ACs or ACs that have linear dependencies (AC-2 depends on AC-1, AC-3 depends on AC-2), when the Recommendation Engine evaluates the project, then it must recommend a `single-session` workflow and not suggest Agent Teams.

**AC-7 — RE includes cost comparison in its output**
When the Recommendation Engine produces an Agent Teams recommendation, the output must include a cost comparison table showing: estimated tokens for single-session, subagents, and the recommended team profile. The cost note must include the disclaimer: "Agent Teams cost 3–4× more than subagents. Use when coordination complexity justifies the premium."

### `/orchestrate` Custom Command

**AC-8 — Feature flag check is mandatory and blocking**
Given that `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` is NOT set in `settings.json` env, when any `/orchestrate` subcommand is invoked (including `/orchestrate status`), then the command must print an error message: `"Error: Agent Teams is not enabled. Set CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 in .claude/settings.json env to activate."` and exit without taking any action. The command must NOT fall back to subagents silently.

**AC-9 — Plan-first gate is enforced by default**
Given that no approved plan exists in the conversation context or `.claude/plan.local.md`, when `/orchestrate {profile}` is invoked without the `--plan-first=false` override, then the command must refuse to create the team and print: `"Error: No approved plan found. Run /plan first, then /orchestrate. Use --plan-first=false to skip this check (not recommended)."` A test invoking with `--plan-first=false` must skip this check and proceed to cost estimation.

**AC-10 — Cost estimate is shown and requires confirmation before team creation**
Given a valid profile and approved plan, when `/orchestrate {profile}` is invoked, then the command must print a cost estimate block in the following format before creating any team state:
```
Teammates: {N}
Estimated iterations: {min}–{max}
Estimated tokens: {token-range}
Estimated cost: ${min}–${max}

Model recommendation: Opus for Lead, Sonnet for Teammates (cost optimization)

Proceed with team creation? (Y/n)
```
If `--dry-run` is passed, the command must print the estimate and exit with code `0` without writing any files or creating any team state.

**AC-11 — Task distribution follows file ownership rules**
Given a PRD with acceptance criteria and a profile's `file_ownership` patterns, when the team is created, then each AC must be assigned to exactly one teammate whose `file_ownership` patterns match the files the AC is expected to touch. ACs with no clear file match must be assigned to the lead and flagged in the task distribution output with: `"⚠️ No file ownership match — assigned to lead for manual distribution."` No two teammates must be assigned the same file path.

**AC-12 — Phase ordering is enforced via `blocked_by` dependencies**
Given a profile with multiple phases (e.g., `fullstack` profile: Database → API+UI → Tests → Review), when tasks are distributed to `tasks.md`, then tasks assigned to phase N must have `blocked_by:` entries referencing the final task of phase N-1. The lead must not dispatch phase N work until the phase N-1 quality gate passes. A test using the `fullstack` profile must verify that the `testing` teammate's first task is blocked by at least one task from the `backend` or `frontend` teammate.

### Hook Recommendations

**AC-13 — TeammateIdle hook checks task completeness and test health**
Given a `TeammateIdle` hook configured in `settings.json`, when a teammate has no remaining tasks assigned and sends an idle signal, then the hook must:
1. Check: are all assigned tasks marked `completed` in `tasks.md`?
2. Check: do tests pass for the files in the teammate's `file_ownership` scope?
3. If either check fails: return exit code `2` with a feedback message sent back to the teammate.
4. If both checks pass: log the idle event to `.claude/logs/team-activity.log` and allow the teammate to sleep.

**AC-14 — TaskCompleted hook validates ACs and test existence**
Given a `TaskCompleted` hook configured in `settings.json`, when a teammate marks a task as completed, then the hook must:
1. Cross-reference the task against the corresponding PRD acceptance criterion.
2. Verify that at least one test file exists that covers the affected files (by checking for `*.test.*` files in related directories or imports).
3. If AC validation fails: return exit code `2` with message `"Task marked complete but AC '{ac_id}' is not demonstrably met. Provide evidence or add a test."`
4. If test existence check fails: return exit code `2` with message `"Task marked complete but no test coverage found for '{file_path}'. Add tests before completing."`

### Documentation & Workflow Examples

**AC-15 — `docs/agent-teams.md` covers the full integration surface**
A documentation file must exist at `docs/agent-teams.md` in the Effectum repo and cover the following topics, each as a named section:
1. **What Agent Teams are** — and when Effectum recommends them vs. subagents vs. single session.
2. **Enabling the feature** — the `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` env var, `settings.json` placement, required Claude Code version (≥v2.1.32).
3. **Agent definition format** — full YAML frontmatter field reference table with type, required/optional, default, and one-line description for each field.
4. **Team profile format** — field reference for `system/teams/*.yaml` files.
5. **The `/orchestrate` command** — all subcommands, options, and the step-by-step execution flow.
6. **Three worked examples**: `web-feature`, `fullstack`, and `review` team workflows with sample invocations and expected outputs.
7. **Cost guidance** — the token/cost formula, how `--max-cost` works, and the rule of thumb for when Agent Teams justify their premium.
8. **Common pitfalls** — the top 5 failure modes from the official docs (lead doing work instead of delegating, file ownership conflicts, too-large tasks, teammate stopping on error, forgetting cleanup).

### 5.2 Recommendation Engine Decision Logic

The RE's existing `recommend(projectConfig)` function must be extended with an `agent-teams` evaluation pass. The decision tree:

```
IF ac_count >= 10 AND domain_count >= 2:
    RECOMMEND agent-teams
    → Select profile: fullstack if database_acs > 0, else web-feature
    → If all ACs are read-only (review/audit): recommend review profile
ELIF ac_count >= 4 AND domain_count == 1 AND parallelizable:
    RECOMMEND subagents
ELSE:
    RECOMMEND single-session
```

Domain detection uses file path pattern matching against AC descriptions:
- `database` domain: ACs mentioning `migration`, `schema`, `RLS`, `supabase`, `prisma`, `seed`
- `frontend` domain: ACs mentioning `component`, `page`, `UI`, `layout`, `style`, `responsive`
- `backend` domain: ACs mentioning `API`, `route`, `service`, `endpoint`, `auth`, `middleware`
- `tests` domain: ACs mentioning `test`, `coverage`, `E2E`, `spec`, `unit`

### 5.3 Hook Script Design

Hook scripts receive context via environment variables set by Claude Code:

**`system/hooks/teammate-idle.sh`:**
```bash
# Environment variables provided by Claude Code:
# TEAMMATE_NAME, TEAMMATE_AGENT, ASSIGNED_TASK_IDS
# Exit 0 = idle OK. Exit 2 = send feedback, keep working.
```

The script must:
1. Read `tasks.md` and check that all task IDs in `$ASSIGNED_TASK_IDS` are marked `DONE`.
2. Run a targeted test command based on `$TEAMMATE_FILE_SCOPE` (extracted from profile).
3. Exit `0` if clean, exit `2` with a descriptive message if either check fails.

**`system/hooks/task-completed.sh`:**
```bash
# Environment variables provided by Claude Code:
# TASK_ID, TASK_DESCRIPTION, AFFECTED_FILES
# Exit 0 = completion accepted. Exit 2 = reject, send feedback.
```

The script must:
1. Cross-reference `$TASK_ID` against the PRD's AC section to verify the task maps to a known AC.
2. Check for test files covering `$AFFECTED_FILES` via glob matching.
3. Exit `0` if valid, exit `2` with specific, actionable feedback if not.

### 5.4 Agent Definition Template Format

Each agent in `.claude/agents/` must use this canonical frontmatter structure:

```markdown
---
name: {agent-role}                    # Required. Lowercase + hyphens.
description: >                        # Required. When to delegate here.
  {One or two sentences. Describe the agent's specialty and the
  triggering condition. This is what Claude reads to decide delegation.}
tools: Read, Glob, Grep, Bash, Edit, Write   # Adjust per role
model: sonnet                         # sonnet | opus | haiku | inherit
permissionMode: acceptEdits           # default | acceptEdits | bypassPermissions | plan
maxTurns: 50                          # Prevent runaway sessions
memory: project                       # user | project | local
isolation: worktree                   # worktree | none (worktree for file-modifying agents)
---

{System prompt. Be specific:}
- What the agent focuses on
- File ownership scope
- Output format expected
- Quality bar expected (e.g., "All functions must have JSDoc")
- What NOT to do (e.g., "Do not modify migration files")
```

### 5.5 Cost Estimation Formula

The formula used in `/orchestrate` Step 3 (from the existing command spec):

```
Per-teammate cost per iteration: ~10,000 tokens (Sonnet)
Lead overhead per iteration:     ~5,000 tokens (Opus)
Total tokens = (teammates × iterations × 10,000) + (iterations × 5,000)
```

**Profile defaults:**

| Profile | Teammates | Min Iter | Max Iter | Min Tokens | Max Tokens | Est. Cost Range |
|---------|-----------|----------|----------|------------|------------|-----------------|
| `web-feature` | 3 | 15 | 40 | 225k | 600k | ~$0.45–$1.20 |
| `fullstack` | 5 | 25 | 60 | 375k | 900k | ~$0.75–$1.80 |
| `frontend-only` | 3 | 10 | 30 | 150k | 450k | ~$0.30–$0.90 |
| `review` | 2 | 5 | 15 | 75k | 225k | ~$0.15–$0.45 |

(Pricing based on Sonnet @ $3/Mtok input, $15/Mtok output; Opus @ $15/Mtok input, $75/Mtok output — blended estimate.)

### 5.7 Worked Examples

#### Example A: Web Feature Team (e-commerce cart)

**Invocation:**
```
/plan Add a shopping cart feature: users can add items, view cart, and checkout.
# ... plan approved ...
/orchestrate web-feature
```

**Team composition:** frontend, backend, testing  
**Phase 1 (backend):** Create `CartItem` table migration, `/api/cart` CRUD routes, cart service  
**Phase 2 (frontend + backend parallel):** CartDrawer component, cart icon in nav, checkout flow  
**Phase 3 (testing):** Unit tests for cart service, E2E for add-to-cart and checkout  

**Expected output:**
```
Team "web-feature-shopping-cart" completed.
Duration: ~42 minutes
Tasks: 12/12
Tokens used: ~280k
Cost: ~$0.84
Quality gates: ✅ Build | ✅ Types | ✅ Tests | ✅ Lint
```

---

#### Example B: Fullstack Feature Team (user notifications)

**Invocation:**
```
/plan Add a notification system: in-app notifications with read/unread state and email digest.
# ... plan approved ...
/orchestrate fullstack
```

**Team composition:** database, frontend, backend, testing, reviewer  
**Phase 1 (database):** `notifications` table, RLS policies, Supabase realtime subscription setup  
**Phase 2 (backend + frontend parallel):** notification API, realtime hook, NotificationBell component, NotificationList page  
**Phase 3 (testing):** Full coverage of notification creation, realtime updates, email digest job  
**Phase 4 (reviewer):** Security audit of RLS policies, notification payload sanitization, XSS review  

**Key pattern:** `database` teammate must complete before `backend` and `frontend` can start (strict dependency on generated types from migration). `reviewer` starts only after all tests pass.

---

#### Example C: Review Team (pre-merge audit)

**Invocation:**
```
/orchestrate review --plan-first=false
```

**Team composition:** quality, security  
**Both teammates are read-only:** They do not modify any files. They produce `review-quality.md` and `review-security.md` reports at the repo root.

**quality teammate prompt:** "Review all changed files (git diff main...HEAD) for code quality, best practices, and performance. Output findings to review-quality.md with severity: critical / major / minor."

**security teammate prompt:** "Review all changed files for security vulnerabilities. Check: SQL injection, XSS, auth bypass, insecure direct object references, RLS policy gaps, hardcoded secrets. Output to review-security.md with severity: critical / high / medium / low."

## 8. Completion Promise

**Agent Teams integration is "done" when:**

> An Effectum user working on a complex, cross-domain feature can type `/orchestrate fullstack` (after an approved `/plan`) and have a 5-teammate team running within 30 seconds — with tasks automatically distributed from the PRD, file ownership enforced, phases ordered correctly, and quality gates validated by hooks.
>
> Before spawning, they see a token/cost estimate and give explicit confirmation. If they just want a code review, they type `/orchestrate review` and get parallel quality + security analysis without touching any code.
>
> The Recommendation Engine tells them *when* to use Agent Teams vs. subagents vs. a single session — including the cost trade-off — so they don't have to decide blindly.
>
> A developer opening the repo for the first time can read `docs/agent-teams.md`, understand the full integration surface, and add a custom team profile without asking questions.

This is Effectum's first multi-agent capability. It makes the framework appropriate for production-grade, multi-layer feature work — not just solo sprints.
