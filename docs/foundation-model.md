# Foundation Model

> `bin/lib/foundation.js`

The Foundation is a fixed, non-negotiable layer of hooks that is installed in **every** Effectum project, regardless of stack, app type, or any other configuration choice. Foundation hooks do not appear in the recommendation flow, cannot be toggled off in manual/customize mode, and are never presented as options during setup.

---

## What Is the Foundation?

When Effectum generates a Claude Code configuration, the output consists of two distinct layers:

```
┌─────────────────────────────────────────────────┐
│  Optional / Recommended Layer                   │
│  (commands, skills, MCPs, subagents, opt. hooks)│
│  ← driven by recommendation engine             │
├─────────────────────────────────────────────────┤
│  Foundation Layer                               │
│  (always-active hooks)                          │
│  ← always present, not configurable            │
└─────────────────────────────────────────────────┘
```

The Foundation layer exists because some behaviours are prerequisites for safe, coherent agent operation. Without them, even a well-configured optional layer can produce destructive or incoherent results. They are not quality-of-life features — they are safeguards.

---

## Foundation Hooks

All eight Foundation hooks are defined in `FOUNDATION_HOOKS` in `foundation.js`. They are exported and injected into the generated configuration by the template engine; they are never part of `HOOK_RULES` (the optional hook list used by the recommendation engine).

### 1. File Protection

**Key:** `file-protection`

Blocks write operations targeting sensitive files and directories:

- `.env` and all `.env.*` variants
- `.git/` and any file inside it
- Dependency lock files (`package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`, `poetry.lock`, `Pipfile.lock`, `Package.resolved`, etc.)
- `secrets/` directory tree

**Why non-negotiable:** Allowing an agent to overwrite `.env` or lock files would expose credentials or silently corrupt the dependency graph. These files must only change via intentional user or package-manager action.

---

### 2. Destructive Command Blocker

**Key:** `destructive-blocker`

Intercepts and blocks shell commands that could cause irreversible, wide-scope damage:

- `rm -rf /` and similar recursive deletions from root
- `DROP TABLE`, `DROP DATABASE`, `TRUNCATE` SQL statements issued outside a migration context
- `git push --force` / `git push -f` against protected branches
- `git reset --hard` without an explicit commit ref

**Why non-negotiable:** A single misfire of these commands can permanently destroy data, history, or entire databases. No recommended workflow should ever need them unconditionally. The blocker forces an explicit override rather than silent execution.

---

### 3. Git Context Loader

**Key:** `git-context`

At the start of every session, injects a structured summary into the agent's context:

- Current branch name
- Uncommitted changes (`git status --short`)
- Last N commit messages and authors

**Why non-negotiable:** Without branch and status awareness, an agent working in an unfamiliar repo is likely to create commits on the wrong branch, duplicate work already committed, or miss unstaged changes that affect compilation. This hook provides the minimum orientation every coding session requires.

---

### 4. Guardrails Injection

**Key:** `guardrails-injection`

Reads `guardrails.md` from the project root (generated by Effectum during setup) and injects its content:

- At session start, before any user message is processed
- After every context compaction event

**Why non-negotiable:** The `guardrails.md` file contains the project's architecture principles, stack-specific constraints, and quality gates (e.g., "never use Pages Router", "pnpm only", "Zod on all boundaries"). Without injection, these constraints are invisible to the agent after context is compacted. Double-injection at compaction is intentional — compaction discards prior system context.

---

### 5. Post-Compaction Context

**Key:** `post-compaction`

Triggered immediately after a context compaction event. Restores:

- The active task or goal state
- Any in-flight tool results that were truncated
- Workspace state summaries

**Why non-negotiable:** Claude Code's compaction mechanism prunes the conversation window to stay within context limits. Without active restoration, the agent continues after compaction with an incomplete picture of what it was doing, leading to repeated work, broken partial operations, or contradictory changes.

---

### 6. Error Logger

**Key:** `error-logger`

Appends a structured JSON record to `.claude/logs/tool-failures.jsonl` whenever a tool call fails. Each record includes:

- Timestamp (ISO 8601)
- Tool name
- Input arguments
- Error message and code

**Why non-negotiable:** Tool failures are silent by default — the agent receives an error string but nothing persists. Without a log, diagnosing intermittent failures (flaky tests, network errors, permission issues) is impossible. The log provides an audit trail that both the user and future agent sessions can inspect.

---

### 7. Transcript Backup

**Key:** `transcript-backup`

Before a context compaction event fires, copies the full session transcript to `.claude/transcripts/` with a timestamped filename.

**Why non-negotiable:** Once compaction discards context, it cannot be recovered from the running session. Backups allow post-hoc debugging ("what did the agent actually do before it went off-track?") and support manual context restoration if needed.

---

### 8. Auto-Formatter

**Key:** `auto-formatter`

After every `Edit` or `Write` tool call, runs the stack's configured formatter on the modified file if it matches the stack's file glob:

| Stack | Formatter | Glob |
|---|---|---|
| `nextjs-supabase` | `npx prettier --write` | `ts\|tsx\|js\|jsx\|json\|css\|md` |
| `python-fastapi` | `ruff format` | `py` |
| `swift-ios` | `swift-format format -i` | `swift` |
| `generic` | *(none)* | `*` |

Formatter configuration is drawn from `FORMATTER_MAP` in `constants.js`.

**Why non-negotiable:** Formatting is the lowest-level quality gate. An agent that writes unformatted code introduces noise into diffs, can cause lint failures on the next CI run, and makes review harder. Running the formatter automatically on every write means formatting is never a separate, forgotten step. Stack-specific guardrails explicitly instruct agents *not* to run the formatter manually precisely because this hook handles it.

---

## Commands Always Active

The Foundation does not define commands — all commands (including the nine core ones) are part of the optional/recommended layer. However, the nine core commands (`plan`, `tdd`, `verify`, `code-review`, `build-fix`, `refactor-clean`, `e2e`, `ralph-loop`, `checkpoint`) carry `always: true` in `COMMAND_RULES`, meaning the recommendation engine always includes them regardless of tag set. They are functionally always present but technically live in the recommendation layer, not the Foundation.

## MCPs Always Active

Similarly, the three core MCP servers (`context7`, `playwright`, `sequential-thinking`) carry `always: true` in `MCP_RULES`. They are always recommended but are part of the recommendation layer, not the Foundation.

## Summary Table

| Hook | Fires On | Blocks / Injects |
|---|---|---|
| File Protection | Every write attempt | `.env`, `.git`, lock files, `secrets/` |
| Destructive Command Blocker | Every bash call | `rm -rf /`, `DROP TABLE`, force push, `reset --hard` |
| Git Context Loader | Session start | Branch, status, recent commits |
| Guardrails Injection | Session start + post-compaction | `guardrails.md` content |
| Post-Compaction Context | Post-compaction | Task state, workspace summary |
| Error Logger | Tool failure | Appends to `.claude/logs/tool-failures.jsonl` |
| Transcript Backup | Pre-compaction | Copies transcript to `.claude/transcripts/` |
| Auto-Formatter | Post Edit/Write | Runs stack formatter on modified file |
