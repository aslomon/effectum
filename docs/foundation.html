System

The Foundation

The Foundation is the always-active layer that runs in every effectum session. It cannot be disabled and does not require configuration. It enforces formatting, quality gates, safety constraints, and session continuity automatically.

CLAUDE.md as source of truth

Everything effectum does is driven by CLAUDE.md. This file is read at the start of every session and defines the stack, quality rules, architecture principles, commit conventions, and available commands. No behavior is hardcoded — it all flows from this single file.

CLAUDE.md configures

  • Primary tech stack and package manager
  • Architecture principles and patterns
  • Code quality rules (max lines, naming, etc.)
  • Quality gate commands (build, test, lint)
  • Commit message conventions
  • Available workflow commands
  • MCP servers and their configuration
  • Autonomy level for Claude decisions

CLAUDE.md in the project root is the local config. The global config at ~/.claude/CLAUDE.md applies to all projects. Project config overrides global config when both are present.

Foundation hooks

These 10 hooks are always active. They run automatically at the right moment — you do not invoke them and you cannot skip them. They exist because ignoring them in past sessions caused real problems.

Auto-format

always on

Fires: After every Edit/Write tool call

Runs Prettier (JS/TS) or Ruff (Python) automatically. You never need to format manually — the hook always fires before Claude moves on.

CHANGELOG auto-update

always on

Fires: At session stop, after meaningful changes

Appends an [Unreleased] entry to CHANGELOG.md summarizing what changed. Never duplicate this manually.

Commit message gate

always on

Fires: Before every git commit

Blocks commits with messages shorter than 10 characters or that are non-descriptive. Enforces conventional commit format.

Stop quality gate

always on

Fires: When Claude attempts to finish a task

Verifies all requested tasks are actually complete before stopping. If ok: false is returned, Claude must continue working.

Subagent quality gate

always on

Fires: When a subagent reports completion

Verifies subagent output is real and complete — not just a report of findings. Subagents must finish their tasks fully.

Error learning

always on

Fires: After any tool failure

Logs failures to .claude/logs/tool-failures.jsonl. At session start, Claude reads this file to avoid repeating past mistakes.

Guardrails injection

always on

Fires: At session start and after context compaction

Loads ~/.claude/guardrails.md (global) and .claude/guardrails.md (project) into context. Guardrails are re-injected after compaction so they are never forgotten.

Transcript backup

always on

Fires: Before context compaction

Saves a copy of the session transcript to .claude/backups/ before compaction. Provides an audit trail and recovery point.

Protected files

always on

Fires: Before any Write tool call

Blocks writes to .env, .env.local, .env.production, secrets/, .git/, and lock files (pnpm-lock.yaml, package-lock.json). These files can only be modified through Bash.

Destructive command blocker

always on

Fires: Before any Bash tool call

Blocks rm -rf /, DROP TABLE, git push --force to main/master, and git reset --hard. These operations require explicit user confirmation.

Foundation hooks cannot be disabled because they are safety nets, not preferences. Disabling the protected files hook would allow accidental .env overwrites. Disabling the destructive command blocker would allow unrecoverable data loss. The Foundation assumes the worst-case scenario so you do not have to.

Guardrails system

Guardrails are curated lessons from past sessions — rules that exist because ignoring them caused problems before. They are injected into every session automatically and re-injected after context compaction so they are never forgotten.

Global guardrails

~/.claude/guardrails.md

Apply to every project you work on. Contain universal rules like never skipping hooks, always using non-interactive CLI flags, and never committing lock files.

Project guardrails

.claude/guardrails.md

Project-specific rules committed with the repo. Contain project-specific things like never calling a deprecated API, always using a specific database client, or RLS on every table.

When you encounter a problem that Claude keeps repeating, add a rule to .claude/guardrails.md. It will be injected at every future session start and after every compaction — permanently fixing the behavior.

Secret detection

The Foundation prevents accidental secret exposure through three mechanisms working together.

Protected file writes

All .env files, .env.local, .env.production, and any file inside secrets/ are write-protected. Claude cannot overwrite them using the Write tool. Direct Bash is required, which forces intentionality.

Pre-commit secret scanning

The commit gate scans staged changes for patterns matching API keys, tokens, passwords, and private keys. Commits are blocked if potential secrets are found.

Guardrails reinforcement

The guardrails file explicitly prohibits reading secrets from the environment and embedding them in source code. This rule is re-injected at every session.

TDD enforcement

The /tdd command enforces the RED-GREEN-REFACTOR cycle. It is not a suggestion — the Foundation hooks verify each phase before allowing Claude to proceed.

  1. RED

    Write failing tests that describe the desired behavior. The test-writer agent generates tests first. Build must pass but tests must fail.

  2. GREEN

    Write the minimum code to make the tests pass. Claude implements only what the tests require — no speculative features.

  3. REFACTOR

    Clean up the implementation without changing behavior. Tests must still pass after refactor. The refactorer agent may run here.

If tests pass in the RED phase (before implementation), Claude stops and flags it — the tests are likely testing the wrong thing or already satisfied by existing code. You must either fix the tests or confirm the behavior already exists.
