Intelligent onboarding

/onboard runs 6 parallel analysis agents on your existing codebase and generates a fully-configured CLAUDE.md, guardrails file, and an initial PRD — then verifies everything works with a self-test loop.

What /onboard does

Running /onboard on an existing project triggers a three-phase process: parallel analysis, configuration generation, and a self-test loop. It works on any codebase — greenfield or legacy.

terminal
/onboard

On a fresh project without an existing CLAUDE.md, use the installer instead: npx @aslomon/effectum. Use /onboard when adopting effectum in an already-running project.

6 parallel analysis agents

All six agents run simultaneously to minimize wait time. Each focuses on a different dimension of the codebase and produces a structured report that feeds the configuration generator.

1

Dependencies agent

Reads: package.json, Cargo.toml, requirements.txt, go.mod, pubspec.yaml

Produces: Technology stack map — languages, frameworks, versions, package manager

2

Structure agent

Reads: Directory tree, entry points, config files

Produces: Architectural pattern (MVC, Clean Architecture, Feature-sliced, Monorepo, etc.)

3

Patterns agent

Reads: Source files across all layers

Produces: Coding conventions — naming style, import order, error handling, async patterns

4

Environment agent

Reads: .env.example, config/, secrets management hints

Produces: Required env vars, secret management approach, deployment targets

5

Tests agent

Reads: Test directories, jest/vitest/pytest config, CI config

Produces: Test framework, coverage setup, test file patterns, missing coverage areas

6

Documentation agent

Reads: README, docs/, inline JSDoc/docstrings, API specs

Produces: Project purpose, onboarding notes, existing guardrails, architecture decisions

The 6 agents are the same ones that run during npx @aslomon/effectum on an existing project. The difference is that /onboard can be re-run at any time to refresh configuration after significant refactors.

Self-test loop

After generating the configuration files, effectum runs 7 verification tests before declaring success. If any test fails, it applies targeted fixes and re-runs — looping until all tests pass or a maximum of 5 iterations is reached.

  1. 1CLAUDE.md exists and is syntactically valid
  2. 2settings.json parses without errors
  3. 3Guardrails file exists at .claude/guardrails.md
  4. 4Quality gates match the detected stack
  5. 5Build command exits with code 0
  6. 6Test command exits with code 0
  7. 7Lint command exits with code 0

Tests 5-7 (build, test, lint commands) are only run if the commands can be detected from the project. If your project has no test setup, that test is marked as skipped rather than failed.

/onboard:review

After onboarding, run /onboard:review to get a detailed audit of the generated configuration — what was detected correctly, what may need manual adjustment, and suggested improvements.

terminal
/onboard:review

Review output includes

  • Detected stack accuracy — what was inferred vs. what you actually use
  • Quality gate coverage — which gates apply to your stack
  • Missing guardrails — common mistakes not yet protected against
  • CLAUDE.md completeness — sections that may need expansion
  • Suggested next /prd:new topics based on your codebase state

How onboarded PRDs work with /prd:update

The initial PRD generated by /onboard describes the current state of your codebase — what exists, how it works, and what needs work. When you run /prd:update on it later, effectum uses delta handoffs to preserve the original analysis while layering in new implementation progress.

1

Initial PRD

Generated by /onboard — captures existing architecture, technical debt, and recommended next steps

2

Development happens

You implement features, fix bugs, run /tdd and /verify cycles

3

/prd:update

Compares current codebase state against the original PRD — adds completed items, updates status, notes drift

4

Delta handoff

Original analysis is preserved as historical context; new sections append without overwriting

The delta handoff model means you can run /onboard again after a major refactor without losing the history of what was originally there. The original PRD becomes an archaeological record.
