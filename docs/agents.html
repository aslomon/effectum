Agent specializations

effectum ships with 19 specialized agents across four categories. Agents are invoked by workflow commands — you never call them directly. Each agent focuses on exactly one responsibility and produces a structured, typed output.

Onboarding agents

Run in parallel during /onboard to analyze an existing codebase from six angles.

onboard-depsOnboarding

Scans all dependency manifests to build a complete map of the technology stack — languages, frameworks, libraries, and their versions.

Triggered by

/onboard, npx @aslomon/effectum on existing projects

Produces

Stack map with detected languages, package manager, framework versions, and direct vs. transitive dependency breakdown

onboard-structureOnboarding

Walks the directory tree to identify the project layout and recognize architectural patterns such as MVC, Clean Architecture, Feature-sliced design, or monorepo structures.

Triggered by

/onboard, npx @aslomon/effectum on existing projects

Produces

Architecture label, directory map, entry point list, and module boundary recommendations

onboard-patternsOnboarding

Reads source files across all layers to detect coding conventions — naming style, import ordering, error handling approach, async patterns, and test structure.

Triggered by

/onboard, npx @aslomon/effectum on existing projects

Produces

Conventions report used to populate the Code Quality Rules section of CLAUDE.md

onboard-envOnboarding

Inspects .env.example, config directories, and infrastructure files to identify required environment variables, secret management approach, and deployment targets.

Triggered by

/onboard, npx @aslomon/effectum on existing projects

Produces

Environment variable inventory, secrets management pattern, and deployment target list

onboard-testsOnboarding

Discovers the test framework, coverage configuration, CI pipeline setup, and existing test patterns — including any missing coverage areas.

Triggered by

/onboard, npx @aslomon/effectum on existing projects

Produces

Test framework config, coverage thresholds, test file naming patterns, and a list of untested modules

onboard-docsOnboarding

Reads README, docs/, inline JSDoc/docstrings, and API specifications to understand project purpose, onboarding notes, and existing architecture decisions.

Triggered by

/onboard, npx @aslomon/effectum on existing projects

Produces

Project summary, existing guardrails, architecture decision records, and initial PRD draft

Quality agents

Activated by workflow commands to audit, fix, and improve code quality.

code-reviewerQuality

Performs a security-focused and code quality audit on changed files — checking for OWASP vulnerabilities, logic bugs, dead code, and architectural violations.

Triggered by

/code-review

Produces

Categorized findings with severity levels (critical, warning, suggestion) and actionable fix guidance

build-fixerQuality

Resolves build and type errors incrementally — fixes one error at a time, re-runs the build, and iterates until the build succeeds or the error is escalated.

Triggered by

/build-fix

Produces

Clean build output; creates a fix log with root causes and the changes applied

refactorerQuality

Identifies and removes dead code, extracts duplicated logic into shared utilities, and improves readability without changing observable behavior.

Triggered by

/refactor-clean

Produces

Refactored source files with a summary of removed code and extracted abstractions

security-auditorQuality

Scans for OWASP Top 10 vulnerabilities, hardcoded secrets, insecure dependencies, and missing input validation across the codebase.

Triggered by

/code-review (security pass), /verify

Produces

Security report with OWASP categories, CVE references where applicable, and remediation steps

coverage-analyzerQuality

Measures test coverage against the configured threshold, identifies uncovered branches, and generates a prioritized list of missing tests.

Triggered by

/verify (coverage gate)

Produces

Coverage report with per-file metrics, uncovered lines, and a ranked list of test gaps to address

Generation agents

Create new artifacts — tests, docs, API specs, migrations, and configs.

test-writerGeneration

Generates test files following the RED-GREEN-REFACTOR cycle. Writes failing tests first, then guides implementation, then refactors for clarity.

Triggered by

/tdd

Produces

Test files colocated with source, covering unit, integration, and edge cases at the configured coverage threshold

e2e-testerGeneration

Generates Playwright end-to-end test suites covering critical user journeys — login flows, form submissions, navigation, and API interactions.

Triggered by

/e2e

Produces

Playwright test files in tests/e2e/, with page object models and reusable fixtures

doc-writerGeneration

Writes and updates documentation — JSDoc/TSDoc comments, README updates, API docs, and architectural decision records.

Triggered by

Automatically after /tdd when files lack documentation

Produces

Inline documentation, updated README sections, and API reference markdown

api-designerGeneration

Designs REST or GraphQL endpoint contracts based on PRD requirements — including request/response shapes, error codes, and OpenAPI spec drafts.

Triggered by

/plan (when PRD includes API work)

Produces

OpenAPI 3.1 spec, TypeScript interface definitions, and implementation scaffolding

db-migratorGeneration

Generates timestamped database migration files from schema changes described in a PRD — supporting PostgreSQL, SQLite, and MySQL dialects.

Triggered by

/plan or /tdd (when PRD includes schema changes)

Produces

Up/down migration SQL files, generated TypeScript types from the new schema, and RLS policy stubs

deploy-configurerGeneration

Generates deployment configuration for the target platform — Vercel, Docker Compose, GitHub Actions CI, or Fly.io — based on the detected stack.

Triggered by

/setup (deployment step), /plan (when PRD includes infra work)

Produces

Platform-specific config files, environment variable checklists, and deployment runbooks

Enforcement agents

Verify compliance with type safety, lint rules, and coverage thresholds.

type-checkerEnforcement

Enforces TypeScript strict mode rules — no implicit any, no type assertions, strict null checks, and correct generic usage.

Triggered by

/verify (type-check gate)

Produces

Type error report with file locations, inferred types, and suggested fixes

lint-enforcerEnforcement

Applies the configured linter (ESLint, Biome, Ruff, etc.) and automatically fixes auto-fixable violations before reporting non-fixable ones.

Triggered by

/verify (lint gate)

Produces

Clean linted source files; residual violations listed with rule names and context

Adding custom agents

Any skill file in .claude/skills/ becomes an agent that effectum can invoke. Custom agents follow the same contract as built-in agents — they receive a typed input and must return a structured output.

terminal
# Create a custom agent skill
mkdir -p .claude/skills

# Example: a custom performance-profiler agent
cat > .claude/skills/performance-profiler.md << 'EOF'
# performance-profiler

## Purpose
Profile runtime performance of API endpoints and identify bottlenecks.

## Input
- List of endpoints to profile
- Request payload samples
- Performance budget thresholds

## Steps
1. Run k6 load test against each endpoint
2. Collect p50, p95, p99 latency metrics
3. Compare against thresholds
4. Identify database query patterns causing slow responses

## Output
- Performance report with per-endpoint metrics
- Prioritized list of bottlenecks
- Recommended fixes with estimated impact
EOF

Skill files are plain Markdown. The headings Purpose, Input, Steps, and Output are a convention, not a schema — effectum reads them as instructions, not as structured data.

Custom agents in .claude/skills/ are committed with the repo. The global skills directory ~/.claude/skills/ is available in all projects.
